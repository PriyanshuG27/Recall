# ADR-002: Reciprocal Rank Fusion (RRF) Hybrid Search

* **Status**: Accepted
* **Deciders**: Core Search Team
* **Date**: 2026-07-04

## Context
Pure vector search excels at conceptual similarity but can miss exact keyword or code snippet matches. Pure keyword search misses semantic context.

## Decision
We implement a two-path hybrid retrieval pipeline in `backend/services/search_service.py` using Reciprocal Rank Fusion (RRF):
1. **Vector Path**: 384-dimensional embedding generated by `sentence-transformers/BAAI/bge-small-en-v1.5` matched via `pgvector` HNSW cosine distance (< 10ms).
2. **Text Path**: Summary matched via `pg_trgm` GIN trigram index (< 5ms).
3. **Fusion**: Scores merged using $RRF = \sum rac{1}{60 + rank}$.

## Alternatives Considered
* **Single Vector Search**: Missed exact technical IDs and proper nouns.
* **Elasticsearch Sidecar**: Added unnecessary operational overhead compared to Neon PostgreSQL extensions.

## Consequences
* **Positive**: High precision and recall across semantic queries and keyword matches.
* **Negative**: Requires both `vector` and `pg_trgm` PostgreSQL extensions.
