
RECALL
Fixed Addiction Architecture & Psychology Design
Revised & Corrected — June 2026
Priyanshu Gumber (PriyanshuG27)

What this document is
This is the corrected version of the Recall addiction architecture. Every issue identified in the senior psychology review has been fixed here — the Monthly Prediction, Living Graph, Pulse Score gate, Thought Compatibility cold-start gap, initial friction problems, insight quality requirements, and the near-miss calibration. Where the original document and this one disagree, this one wins.

Part 0 — What Was Broken and What Changed
The original addiction architecture was sound in direction but had five specific mechanical failures that would have killed retention before any mechanic could activate. This part documents each failure, the psychological reason it was a failure, and the exact fix applied in this document.

0.1 Monthly Prediction — Too Vague, Feedback Loop Too Long
Original problem
"In 2-3 weeks you're going to start saving things about X" has a 2-3 week verification window. If the prediction is wrong, users don't check back to disprove it — they forget it. The cognitive splinter mechanism requires a short enough loop that the user can verify or disprove the prediction while the memory is still active.

Psychology: The cognitive splinter effect — an unverified prediction about yourself that sits in working memory demanding resolution — only functions within the timescale of working memory persistence. Predictions with a 2-3 week window are forgotten before they can resolve, which eliminates the mechanism entirely.

Fix applied in this document
Prediction window shortened to 5-7 days. Specificity bar raised — predictions must name a specific tension or cluster-pair trajectory, not a topic category. Three user reactions are all valid return triggers: "that's wrong" → returns to disprove, "how does it know?" → returns out of curiosity, "it's right" → shares immediately. None of these reactions are possible if the prediction is forgotten.

Updated Monthly Prediction Spec
Fires monthly. Window: 5-7 days from send, not 2-3 weeks. Specificity requirement: must reference at least two existing clusters and name the tension between them. Confidence floor: only fire if the trajectory confidence score is above 0.72 based on the last 14 days of graph movement. If confidence is below threshold, skip the month rather than fire a low-quality prediction.

Message template:
Your graph has been building toward something. Based on where [Cluster A] and [Cluster B] have been moving this month, in the next 5-7 days you are likely to save something about [specific tension]. Not a topic — a tension. The one between [named contradiction]. Check back in a week.

Constraints: Never use hedging language ("might", "possibly", "perhaps"). State it as a confident read. If it fires and it's wrong, the user corrects it — and that correction is itself high-value metadata about what the graph misread. A wrong confident prediction that gets corrected teaches the system more than a vague prediction that gets ignored.

0.2 Living Graph — Visual Decay Alone Does Not Drive Behavior
Original problem
The Living Graph mechanic assumed users would open Recall to "tend to" the graph like a plant based on visual node cooling. There is no evidence from comparable products that visual decay drives return visits without an accompanying behavioral trigger. The visual alone is decoration without a message.

Psychology: Visual state change requires a notification bridge to convert into behavior. The cooling graph is a reward for users already inside the product. It does nothing for return visits because users who have drifted are not inside the product to see it. This is the core mechanical failure — the mechanic was designed entirely for retained users and does nothing for lapsed ones.

Fix applied in this document
Visual node temperature is kept as an in-product experience. A new Telegram trigger fires when three or more clusters cross the cooling threshold in the same 48-hour window. The trigger converts the visual state into a behavioral pull for lapsed users. The message names what is cooling specifically — not a generic "your graph needs attention" push.

Living Graph — Two-Layer Implementation
Layer 1 — In-product visual (unchanged from original spec):
Nodes warm when recently connected, cool over 5-7 days of inactivity
Cluster temperature computed from last_active_at on semantic_hubs
Visual: warm clusters glow amber, cooling clusters shift toward gray, cool clusters visually separate from the main graph
This layer rewards users who are already inside the product — it is not a return mechanic

Layer 2 — Telegram trigger (new, required for behavioral impact):
Fires when: 3+ clusters cross below 40% temperature in the same 48h window AND user has not opened the frontend in 72+ hours
Message template (bold indicates required field):
"Three things your graph was tracking went quiet in the last two days. [Cluster A], [Cluster B], and [Cluster C]. Not gone — just cooling. Your graph remembers them. Whether you do is the question."
Frequency cap: maximum once per 10 days per user. Cooling messages sent more often than this convert from presence to nagging.
Never fires if the user has been active in the last 72 hours — this is a lapse trigger, not a daily push

0.3 Pulse Score — Gated Too Late to Drive Retention
Original problem
Pulse Score activating at 100 nodes means it only appears after retention is already solved. By the time a user has 100 nodes, they are retained — Pulse did not get them there. As a cold-start or mid-funnel mechanic it contributes nothing. The gate makes Pulse a reward for already-retained users rather than a retention driver.

Psychology: Progress visibility drives motivation only when it appears early enough for the user to form a mental model of what they are building toward. A metric that appears after 100 saves cannot shape the mental model during the critical first 2 weeks when retention decisions are made. It needs to be visible from session one — growing from a small number — to function as the intended identity metric.

Fix applied in this document
Pulse Score visible from node 1, starting small and scaling with graph growth. The 100-node gate is removed. A new milestone system replaces it: key thresholds (15, 30, 50, 100, 200 nodes) trigger capability unlocks that are announced with specificity, not just a number changing. Pulse becoming a meaningful metric is a natural consequence of graph growth, not a locked gate.

Updated Pulse Score System
Early Pulse (nodes 1-14): Score computed but displayed with explicit framing — "Your Pulse is building. At 15 nodes, your Mind Type unlocks." The number is visible but contextualized as early-stage. This converts Pulse from a meaningless small number into a progress signal toward a meaningful unlock.

Milestone unlock table (replaces the 100-node Pulse gate):

Node threshold
Unlock + announcement
5 nodes
First Pattern Report fires. "Here is what your mind has been working on."
15 nodes
Mind Type unlocks. Computed from graph structure, not a quiz. Announced with full specificity.
30 nodes
Monthly Prediction activates. First prediction fires within 48 hours of hitting the threshold.
50 nodes
Thought Compatibility unlocks. Includes the fast friend onboarding flow (see Part 0.4).
100 nodes
Pulse Score becomes a ranked metric. User can see where they sit relative to graph density benchmarks.
200 nodes
Public Graph unlocks. One link. Their mind, visible to anyone.

Each threshold fires a specific Telegram message naming the new capability. The message is not a notification — it reads as an observation. "Your graph just crossed 15 nodes. Something can be computed now that couldn't be before."

0.4 Thought Compatibility — Missing the Fast Friend Onboarding Flow
Original problem
Thought Compatibility depends on two users connecting their Recalls simultaneously. Most people who want to use it with a friend have a friend who does not yet have Recall. The share → wait → friend sets up → friend builds enough graph to compare cycle takes days and the excitement dies in the gap. There is no same-conversation onboarding path.

Psychology: Interpersonal self-disclosure escalation — the mechanism that makes Thought Compatibility sticky — works only when the disclosure happens in a live conversation, not days later after setup friction. The product needs to be designed so that the person receiving a compatibility share can generate a meaningful partial result within one Telegram session, without days of prior saving.

Fix applied in this document
A Friend Fast-Track flow activates when a user shares a Thought Compatibility invite. The friend is walked through a 5-question seeding sequence (not screenshots, not links — direct Telegram replies) that generates enough graph signal for a meaningful compatibility output within a single 10-15 minute session. The comparison fires within the same conversation the invite was shared in.

Friend Fast-Track Flow
Trigger: User A generates a Thought Compatibility invite link. Friend B opens the link in Telegram.

Friend B session (5 questions, all Telegram-native):
Q1: "What's one thing you've been thinking about for weeks but haven't told anyone?" → tagged as cluster seed, type: preoccupation
Q2: "What's something you changed your mind about in the last year?" → tagged as cluster seed, type: belief shift
Q3: "What do you read that you'd be embarrassed to admit?" → tagged as cluster seed, type: hidden interest
Q4: "What problem keeps showing up in your life in different forms?" → tagged as cluster seed, type: recurring tension
Q5: "What do you think about right before you fall asleep?" → tagged as cluster seed, type: unresolved loop

Each answer is processed through the AI cascade as a context-rich node. 5 nodes with this level of metadata is enough for a meaningful compatibility comparison because the seeding questions are specifically designed to surface the cluster-types the compatibility algorithm needs — preoccupations, tensions, hidden interests, belief shifts.

Output fires within the same session, maximum 15 minutes after Friend B sends Q5. The comparison report uses the standard Thought Compatibility format with the full Mind Collision output. Friend B receives an invite to continue building their own graph after the report fires — the comparison is the hook, their own graph is the retention mechanic.

Implementation note: Friend B's 5 seeded nodes are created in a temporary user account tied to their Telegram ID. If they continue, the account converts to a full account. If they do not, the data is retained for 30 days and then purged. Never auto-create a full account without explicit continuation — dark patterns break trust in exactly the introspective product category where trust is the entire moat.

Part 1 — Fixing Initial Friction (Complete Revised Spec)
Every issue identified in the friction analysis is addressed here with specific implementation guidance. The five friction points are ordered by severity.

1.1 Friction Point 1 — Zero-Value First Session (Critical)
This is the only friction point that, if not fixed, makes every other fix irrelevant. A user who leaves the first session without a magical moment will not return.

First-Session Magic — Revised Implementation
The original spec describes first-session magic correctly in concept but does not give enough operational detail to guarantee it fires reliably. The following is the implementation spec that makes it reliable.

Step 1 — Minimum viable batch:
Ask for 5-10 screenshots from WhatsApp Saved Messages during onboarding. Exact copy: "Screenshot 5 to 10 things from your saved messages — doesn't matter what they are. This is just to get your graph started."
Low-stakes framing is required. "Doesn't matter what they are" reduces perceived judgment on input quality, which is the primary reason users abandon before completing the batch send.

Step 2 — Debounce before processing (prerequisite — see Part 1.4):
Redis debounce window must be active before this flow or the batch produces 10 noise nodes instead of 1-3 signal clusters
Do not ship the onboarding flow without the batching fix live

Step 3 — First connection algorithm:
After batch processes, run pairwise cosine similarity across all new nodes
Primary threshold: 0.75+ → confirmed connection. Fire the magical moment message.
Fallback threshold: if no pair reaches 0.75, lower to 0.60 and flag as tentative in the message copy. "Your graph found something — it's early, only two data points, but these two things might be related."
If nothing reaches 0.60: do not send silence. Send: "Your graph is seeded. I'll be watching for the first connection — usually shows up within the first few saves." This is honest, not a failure message.

Step 4 — Message specificity requirement:
The connection message must name the literal saved items, not paraphrase them into abstraction
Correct: "Something about Kobe Bryant and something about the Feynman technique — both about mastery through repetition."
Wrong: "You seem to be interested in learning and performance."
If the message would survive with a different user's items substituted in, it has failed the specificity test. Reject it and regenerate.

1.2 Friction Point 2 — The Ask Before the Give (High)
Every addiction mechanic activates only after the user has a saving habit. The Day 1-5 onboarding sequence exists to manufacture that habit before requiring it.

Day 1-5 Onboarding Sequence — Full Revised Spec

Day 0 (same day as first batch, a few hours later):
Message: "I looked at the thing you sent me. Quick question — was this for you, or were you planning to act on it or share it with someone?"
Maps to the exact item_id of the first processed item. Uses the pending_context:{chat_id} → item_id Redis key. TTL: 6 hours.
Any reply writes to that item's context_note column. Does not create a new node.

Day 1 (next morning):
Message: "Still thinking about [literal thing they saved]? I'm not doing anything with it yet — just checking if it's still on your mind, or if it's already done its job."
Same mapping rule. Same item_id. Any reply goes to context_note, not a new node.
Silence is logged as a signal, not a failure. Do not escalate based on silence.

Day 2-3 (only if Day 0 or Day 1 got a reply):
Message: "What's one thing you've been meaning to look into but haven't saved anywhere yet?"
This is explicitly a new-node fishing message. Any reply goes through the normal save path — new row, AI cascade, embedding. Do not write to the existing item's context_note.
Hard stop: if neither Day 0 nor Day 1 got a reply, do not send this message. Silence after two messages means the product hasn't earned a third unprompted contact.

Critical implementation boundary
The Day 0/1 messages map back to existing items. The Day 2-3 message creates new items. Conflating these two behaviors silently corrupts either the context metadata or the node count. Build them as two explicitly different code paths with the distinction enforced at the data layer, not the application layer.

1.3 Friction Point 3 — Context Prompt Feels Like a Form (Medium)
The one-sentence context prompt is load-bearing. Without context metadata, insight quality collapses. But if it reads as a mandatory field, users ignore it, which has the same effect as not having it.

Context Prompt — Revised Phrasing System
Replace the single fixed phrasing with a rotation pool of 8 variants. Each variant signals genuine curiosity rather than data collection. All variants are skippable — no follow-up if the user ignores it.

Variant
Message copy
Curiosity
"This one caught my eye — what made you save it?"
Timing
"What was happening when you found this?"
Future
"What are you planning to do with this, if anything?"
Friction
"Is this something you agree with or something you're not sure about?"
Identity
"Is this how you think or how you want to think?"
Connection
"Does this connect to anything else you've been thinking about?"
Stakes
"Is this urgent or just interesting?"
Surprise
"Did this confirm something or surprise you?"

Rotation logic: never repeat the same variant within 5 saves. Track last variant used per user in Redis. Weight toward variants that have historically produced longer replies for that user — longer replies produce better context metadata.

Skip handling: if a user ignores the context prompt 3 times in a row, pause it for 5 saves and resume. Repeated ignoring signals the prompt feels like friction for that user. Resuming after a gap resets the expectation.

1.4 Friction Point 4 — Screenshot Batching Creates Phantom Nodes (Medium)
Redis Debounce Fix — Implementation Spec
This is an engineering bug, not a UX problem. The fix is required before any cold-start onboarding flow ships.

On each incoming Telegram message or photo, push the payload to Redis list batch:{chat_id} and (re)set a 5-second expiry
Only when the 5-second window closes with no new arrival does the real ingest task push to recall:tasks — carrying the whole accumulated batch as one task
The worker runs OCR on every image in the batch first, concatenates extracted text in arrival order, then makes one AI cascade call: "Is this one coherent piece of content split across images, or several unrelated things bundled together?" The model returns 1 summary or N grouped summaries — never one output per image by default
Why this reduces load: 8 screenshots = 8 worker slots before the fix. After the fix = 1 slot for one larger OCR+LLM call. The 5-second debounce trades a flat 5-second delay (invisible — user is not watching for a reply) for both correctness and reduced concurrency pressure on Semaphore(3)
The existing 50ms Telegram webhook ACK is unaffected — that fires immediately as always

Do not ship cold-start onboarding without this fix live
The onboarding flow specifically asks users to send a batch of 5-10 screenshots. Without the debounce, that batch becomes 10 unrelated nodes. The first-session magic algorithm runs on 10 noise nodes and either fails to find a connection or finds a spurious one. The first impression is poisoned at exactly the moment it matters most.

1.5 Friction Point 5 — Chrome Extension Discoverability (Lower)
The Chrome extension exists but is not surfaced prominently in the onboarding flow for non-Telegram-native users. This is the primary bridge for users who live in a browser rather than Telegram.
Add Chrome extension install link to the onboarding message sequence at Day 0, framed as an alternative save path: "If you're usually on your laptop, the Chrome extension lets you save directly from any webpage — one click, no copy-paste."
Do not frame it as a replacement for Telegram — frame it as a complement. Telegram remains the primary interface. The extension is for capture-in-context.

Part 2 — Near-Miss Mechanic (Missing From Original Spec)
The near-miss mechanic was identified as the highest-leverage missing piece in the original addiction architecture review. It converts the user from passive recipient ("the graph showed me something") to active participant ("I'm trying to close a gap I can see"). This is the difference between an app people appreciate and an app people tend to.

Why this is more powerful than a confirmed connection
A confirmed connection rewards. A near-miss activates. The Wordle and crossword pull harder than a quiz with the answer shown immediately because the gap between current state and closure creates agency. Closing a near-miss means the user engaged with an idea — which is a good outcome regardless of whether they return tomorrow. A confirmed connection does not create that engagement.

2.1 Near-Miss Detection Spec
Near-miss detection runs as part of the unified Insight Candidate Pipeline (nightly, extending the existing louvain_clustering cron). It does not require a separate scan.

Threshold band: 0.71-0.75 cosine similarity (below confirmed-connection floor of 0.75)
Pairs below 0.71 are discarded — too far from connection to be meaningful signal
Pairs above 0.75 are promoted to confirmed connections — handled by the Recall Moment / Cluster Portrait mechanics
Near-miss status persists for 72 hours in the insight_candidates table with status: near_miss

2.2 Calibration — Critical
The threshold band is a tunable parameter, not a fixed value. The numbers above (0.71-0.75) are starting points. The correct values must be calibrated empirically against real graph data.

Calibration failure modes:
Too wide (e.g., 0.55-0.75): Most cluster pairs qualify. User gets a near-miss message almost every day. Stops feeling like signal, starts feeling like noise. Users mute within 2 weeks. This is fatal.
Too narrow (e.g., 0.73-0.75): Fires rarely. Ineffective but survivable — the mechanic just doesn't contribute much.

Calibration target: 1-2 genuine near-misses per user per week across a typical 40-node graph. Recalibrate per-user as graph density grows — a 200-node graph will have more natural near-miss candidates and needs a tighter band.

Recalibration procedure: after 4 weeks of data, compute near-miss message send rate per user. Users receiving 5+ near-miss messages per week → tighten their band by 0.02. Users receiving 0 → widen by 0.02. Never share the band value with the user. Never round up to imply a connection exists when one does not.

2.3 Near-Miss Message Template
"Your graph almost made a connection. Something from [N] days ago is sitting at [X]% similarity with [Cluster Name]. One more save in this space and it might cross."
Constraints:
Always state the literal percentage or a plain-language equivalent. Never round up.
"71% similar" is acceptable. "Very close" without a number is not — it loses the specificity that makes the mechanic credible.
"Might cross" is correct. "Will cross" is a lie — do not promise what the algorithm cannot guarantee.

Part 3 — Returning Themes Mechanic (New, Not in Original Spec)
The original insight pipeline includes a 60-day novelty filter that discards cluster-pairings that have fired before. This is correct for preventing spam. But it has an unintended side effect: it silently discards the most valuable signal in the entire graph.

The insight the 60-day filter was deleting
When the same cross-cluster pair resurfaces after 60+ days, that is not a duplicate — it is evidence of a deep, recurring preoccupation. The user's mind returned to the same tension independently, across time, without being prompted. This is the deepest form of self-knowledge Recall can surface.

3.1 Returning Themes Detection
A cluster-pair that fires, enters the 60-day suppression window, and then re-emerges after day 60 is tagged as status: returning_theme in the insight_candidates table
Returning themes are routed to a separate message template — distinct from new connections
Frequency: maximum once per returning theme per 90-day window. If the same pair keeps recurring, it should eventually become a named cluster — the Louvain clustering job should be evaluated to see why it has not already merged them

3.2 Returning Themes Message Template
"This connection showed up again. [Cluster A] and [Cluster B] — your graph saw this eight weeks ago. It's still true. Your mind has been working on this longer than you realized."
This message type lands differently from a new connection because it implies continuity of self across time — one of the most psychologically resonant signals available. It tells the user their mind has been preoccupied with something without their conscious awareness. That is the product's deepest value proposition, delivered.

Part 4 — Insight Quality Is Not a Feature. It Is the Product.
The one thing that cannot be iterated toward
Every mechanic in this document is a delivery wrapper around one capability: can the AI cascade reliably produce a sentence about a specific person's saved items that reads as 'no one but me could have written this about me'? If yes, every mechanic works. If no, every mechanic is decoration on a broken core. Shipping user-facing mechanics before this capability is proven burns user trust permanently. There is no second chance at first impression for an introspective product.

4.1 The Specificity Test
Before any insight message ships to a user, it must pass this test: would this message still make sense if you substituted a different user's saved items?
If yes → the message has failed. Reject and regenerate.
If no → the message is grounded. It can ship.

This test is not a vibe check. It is a literal substitution test that should be run programmatically in the evaluation loop as a guard before the message enters the delivery queue.

4.2 The Tensions-Over-Traits Principle
Every insight generation prompt in the system must be optimized to find tension-pairs across clusters, not single-cluster summaries.

Trait insight (weak)
Tension insight (strong)
"You're interested in self-improvement"
"You save discipline content right after freedom content, every single time"
"You think a lot about leadership"
"You save things about power right after things about vulnerability — never in the other order"
"You're a systems thinker"
"You save systems thinking content when things feel out of control. Your graph has been tracking this for six weeks."
"You care about learning"
"You save Feynman and you save Kobe. You've been trying to solve the same problem about mastery since September."

4.3 The Hard Boundary
Non-negotiable — not a style choice
No insight may imply a clinical or diagnostic read of the user's mental state. "You're trying to resolve a guilt loop" sits on the wrong side of this line. Keep every insight at the level of 'here is a pattern in what you saved,' never 'here is what's actually going on with you psychologically.' An app that narrates a user's subconscious motivations with high authority is risky for anyone in a fragile state. This risk grows as insight quality improves — not shrinks.

4.4 The Banned Phrase List
Any generated message containing one of these patterns is an automatic reject, regardless of other quality:
"You seem interested in..." / "You have a passion for..." — trait language, not tension language
"This might suggest..." / "It's possible that..." / "Perhaps you..." — hedging that undercuts calibrated confidence
"Your journey" / "your growth" / "your path" — generic self-help register
Any sentence that would survive substitution with a different user's items
"This reveals that you're someone who..." followed by a personality trait

Part 5 — Corrected Build Order
This is the order that creates the retention hook as fast as possible, with all dependencies respected.

Priority

1 — Ship first
Redis debounce + batching fix. Prerequisite for everything. Cold-start flow must not ship without this.
2 — Ship first

3 — Core hook
First-session magic. The hook. Nothing else matters until this reliably produces a specific, non-substitutable connection message.
4 — Cold start
Day 1-5 onboarding sequence. Gets new users to 5+ nodes inside days instead of indefinite self-motivated weeks.
5 — Insight quality
Evaluation loop + specificity test. Build and prove before any mechanic ships on top of it. Cannot be iterated toward after the fact.

Morning Mystery + Evening Answer. Builds the daily check-in habit. 3 consecutive days = habit forming.

Unified Insight Candidate Pipeline. Single nightly job backing all downstream mechanics. Build once, not four times.

Living Graph Telegram trigger (Layer 2). Visual decay was already built — this adds the behavioral bridge for lapsed users.
